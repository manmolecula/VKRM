# Практическая реализация методики миграции Vue.js приложения

Данный репозиторий содержит практическую часть выпускной квалификационной работы на тему: **«Разработка методики миграции и рефакторинга высоконагруженного веб-приложения на Vue.js с переходом к современной архитектуре на основе композиционных функций»**.

В рамках работы проведен сравнительный анализ двух подходов к разработке на примере реальных компонентов системы различного уровня сложности (исходный код анонимизирован).

## Структура проекта

### Пример 1: Компонент работы со списками (Базовый уровень)
Демонстрация рефакторинга стандартного компонента с сетевой загрузкой и пагинацией.
* `src/SingleComponent.vue` — Исходная реализация (Vue 2, Options API). Логика работы с сервером и состояние формы тесно связаны в одном модуле.
* `src/MigratedSingleComponent/` — Рефакторинг (Vue 3, Composition API).
  * `MigratedSingleComponent.vue` — Компонент представления.
  * `composables/useMigratedSingleComponentLogic.js` — Изолированная логика запросов и вычисления диапазона страниц.

### Пример 2: Агрегирующий дашборд (Сложный уровень)
Демонстрация применения методики к компоненту с нарушением принципа единой ответственности (смешение управления интерфейсом, бизнес-правил и глобального состояния).
* `src/MultiComponent.vue` — Исходная реализация (Vue 2). Характеризуется жесткой связностью с хранилищем Vuex и зависимостью от глобального объекта `window`.
* `src/MigratedMultiComponent/` — Рефакторинг (Vue 3).
  * `MigratedMultiComponent.vue` — Компонент, зависящий только от передаваемых данных.
  * `composables/useMigratedMultiComponent.js` — Бизнес-логика контроля доступа и фильтрации, полностью независимая от фреймворка.

* `tests/` — Набор сравнительных модульных тестов.

## Запуск проекта и тестов

1. Установка зависимостей:
```bash
npm install
```

2. Запуск тестов:
```bash
npm test
```

## Результаты миграции и количественные метрики

Основной фокус при оценке результатов направлен на сравнение стоимости поддержки и_verификации_ кода, так как архитектурный рефакторинг нацелен на снижение когнитивной нагрузки и изоляцию бизнес-правил.

### 1. Базовый компонент (списки)
В ходе миграции применена стратегия извлечения бизнес-логики и оптимизация подсистемы реактивности.

| Метрика | Vue 2 (Options API) | Vue 3 (Composition API) | Динамика |
|---------|---------------------|-------------------------|----------|
| Объем кода тестов | 68 строк | 44 строки | Сокращение на 35% |
| Время выполнения тестов | ~13 мс | ~3 мс | Ускорение в 4.3 раза |
| Необходимые зависимости для тестов | Инициализация компонента, 12 имитаций UI-элементов | Только подмена сетевого запроса | Снижение связанности |
| Обработка больших массивов | Глубокая реактивность (Observer) | Оптимизированная (`shallowRef`) | Снижение нагрузки на память |

### 2. Сложный агрегирующий компонент (дашборд)
Демонстрация того, как методика справляется с компонентами, имеющими сложную инфраструктурную связанность (Vuex, глобальные переменные).

| Метрика | Vue 2 (Монолит) | Vue 3 (Модульная архитектура) | Динамика |
|---------|---------------------|-------------------------|----------|
| Объем кода тестов | 75 строк | 20 строк | Сокращение в 3.7 раза |
| Время выполнения тестов | ~14 мс | ~2 мс | Ускорение в 7 раз |
| Требования к инфраструктуре тестов | Эмуляция хранилища Vuex, подмена `window`, 22 имитации дочерних компонентов | **0 внешних зависимостей** (вызов чистой функции) | Полная изоляция логики |
| Передача данных в логику | Чтение напрямую из глобального контекста | Через параметры функции (внедрение зависимостей) | Повышение переиспользуемости |

## Анализ качества тестирования

Сравнение проводилось на идентичных сценариях: фильтрация табличных данных и проверка прав доступа (определение визуального состояния элемента по статусу пользователя).

### Проблемы верификации в исходной архитектуре (Vue 2)
Для проверки элементарного правила определения доступа требовалась комплексная подготовка инфраструктуры:
1. **Инициализация хранилища:** Необходимость создания полноценного экземпляра Vuex с описанием модулей, даже если проверяемое правило не взаимодействует со стором напрямую.
2. **Зависимость от среды:** Подмена глобального объекта `window`, что снижает надежность тестов при параллельном запуске и усложняет их поддержку.
3. **Избыточная инициализация:** Использование инструмента `shallowMount` и обязательное описание более 20 компонентов-заглушек (имитаций таблиц, диалоговых окон), которые не участвуют в проверке, но необходимы для парсинга шаблона.

### Преимущества верификации после рефакторинга (Vue 3)
Разделение ответственности позволило вынести проверяемые правила в независимые функции (`use...`).
* **Скорость:** Время выполнения сократилось до 2–3 миллисекунд за счет исключения этапов инициализации виртуального DOM.
* **Изоляция:** Данные для проверки (например, статус авторизации) передаются в функцию как аргумент. Тест полностью абстрагирован от среды выполнения (браузер/сервер).
* **Читаемость:** Тестовый сценарий сводится к проверке возвращаемого значения функции, что соответствует классическим принципам модульного тестирования.

*Примечание: Отказ от использования искусственных задержек (таких как `setTimeout`) в пользу нативного ожидания асинхронных операций позволил получить объективные метрики времени выполнения (~13-14мс для Vue 2). Это время отражает базовые затраты на инициализацию тестовой среды и компонентов фреймворка.*

## Архитектурные выводы

Примененная методика поэтапной миграции позволила достичь следующих результатов:

1. **Снижение когнитивной сложности:** Разделение модуля на уровень представления и уровень логики. Разработчику достаточно изучить функцию composable для понимания бизнес-правил, не анализируя шаблон компонента.
2. **Оптимизация производительности:** Применение механизма `shallowRef` для хранения больших массивов данных исключило избыточное рекурсивное отслеживание изменений внутри элементов списков, что снижает нагрузку на процессор и сборщик мусора.
3. **Устранение жесткой связанности:** Компоненты больше не зависят от конкретной структуры глобальных переменных или модулей состояния (Vuex). Передача конфигурации через аргументы делает логику переносимой между различными частями приложения.
4. **Повышение тестируемости:** Методика перевела процесс проверки бизнес-логики из разряда интеграционных тестов (требующих подъема инфраструктуры) в разряд быстрых модульных тестов. Это гарантирует отсутствие регрессий при дальнейшей поддержке системы.
