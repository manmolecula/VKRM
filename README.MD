# Практическая реализация методики миграции Vue.js приложения

Данный репозиторий содержит практическую часть выпускной квалификационной работы (ВКР) на тему: **«Разработка методики миграции и рефакторинга высоконагруженного веб-приложения на Vue.js с переходом к современной архитектуре на основе композиционных функций»**.

В рамках работы проведен сравнительный анализ двух подходов к разработке на примере реальных компонентов системы разного уровня сложности (анонимизированный код).

## 📁 Структура проекта

### Уровень 1: Компонент списков (Базовая миграция)
- 📂 `src/SingleComponent.vue` — Реализация **"ДО"** (Vue 2 Options API). Монолитный компонент со смешением состояния формы и пагинации.
- 📂 `src/MigratedSingleComponent/` — Реализация **"ПОСЛЕ"** (Vue 3 Composition API).
  - `MigratedSingleComponent.vue` — Тонкий слой представления (View).
  - `composables/useMigratedSingleComponentLogic.js` — Изолированная логика работы с сетью и пагинацией.

### Уровень 2: Сложный дашборд (Масштабирование методики)
- 📂 `src/MultiComponent.vue` — Реализация **"ДО"** (Vue 2). Антипаттерн "God Object" (450+ строк). Сильная связность с Vuex и глобальным объектом `window`.
- 📂 `src/MigratedMultiComponent/` — Реализация **"ПОСЛЕ"** (Vue 3). 
  - `MigratedMultiComponent.vue` — Компонент, зависящий только от внедренных данных (Dependency Injection).
  - `composables/useMultiComponentLogic.js` — Бизнес-логика управления доступами и фильтрацией, не зависящая от фреймворка.

- 📂 `tests/` — Сравнительные тест-сьюты.

| Метрика | Vue 2 (Options API) | Vue 3 (Composition API) | Динамика |

| Метрика | Vue 2 (God Object) | Vue 3 (Composable) | Динамика |
|---------|---------------------|-------------------------|----------|
| Объем кода (View + Logic) | ~380 строк | ~165 строк | Сокращение на **56%** |
| Объем кода тестов | ~75 строк (из них 40 — boilerplate) | ~20 строк | Сокращение в **3.7 раза** |

## 🧪 Анализ качества тестирования

Сравнение проводилось на идентичных бизнес-сценариях: фильтрация данных и проверка прав доступа (Access Control).

### Проблемы тестирования в Options API (Vue 2)
Для проверки простейшего правила (например, смена цвета кнопки в зависимости от статуса пользователя) требовалось:
1. **Создание инфраструктуры:** Инициализация полноценного экземпляра Vuex Store с неймспейсами (даже если логика к нему не привязана напрямую).
2. **Глобальное загрязнение:** Мокирование объекта `global.window`, что делает тесты хрупкими (flaky) при параллельном запуске.
3. **Избыточный рендеринг:** Использование `shallowMount` и прописывание 20+ заглушек (stubs) для UI-компонентов (таблицы, модалки), которые не участвуют в проверяемом сценарии.

### Преимущества тестирования в Composition API (Vue 3)
Благодаря разделению ответственности (Separation of Concerns), бизнес-логика была вынесена в функции `use...`.
* **Скорость:** Выполнение тестов занимает 2-3 миллисекунды, так как отсутствуют накладные расходы на виртуальный DOM.
* **Изоляция (Dependency Injection):** Данные пользователя передаются в функцию как аргумент `initialState`. Тест больше не зависит от среды выполнения (browser/node global context).
* **Читаемость:** Тест представляет собой проверку обычной JavaScript-функции, что снижает порог входа для разработчиков.

> *Примечание к метрикам времени:* В связи с отказом от антипаттерна `setTimeout` в пользу утилиты `flushPromises` (для честности замеров), абсолютное время выполнения Vue 2 тестов составило ~13-14мс. Это время занимает инициализация самого тестового фреймворка (Vitest) и создание обертки компонента (`shallowMount`). Однако на масштабах приложения (сотни компонентов) разница в 7-10 раз на каждом тесте дает кумулятивный эффект, сокращая время прогона CI/CD пайплайнов с минут до секунд.

## 🏗 Архитектурные выводы

Разработанная методика поэтапной миграции позволила:

1. **Снизить когнитивную сложность:** Разделение файла на View (шаблон) и Logic (composable) сократило объем читаемого кода на 27-56%. Разработчику больше не нужно анализировать 20+ полей `data()` для понимания работы одной кнопки.
2. **Оптимизировать производительность:** Применение `shallowRef` для больших списков данных исключило избыточное рекурсивное наблюдение (deep observation) за элементами массивов, снижая нагрузку на сборщик мусора.
3. **Устранить жесткую связность (Tight Coupling):** Компоненты больше не зависят напрямую от глобальных переменных (`window`) или конкретной структуры Vuex-модулей. Передача конфигурации через аргументы делает логику переносимой.
4. **Повысить тестируемость:** Методика превратила интеграционные тесты (требующие поднятия инфраструктуры) в быстрые модульные тесты чистых функций, что гарантированно предотвращает регрессии при дальнейшем рефакторинге.
