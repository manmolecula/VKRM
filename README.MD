# Практическая реализация методики миграции Vue.js приложения

Данный репозиторий содержит практическую часть выпускной квалификационной работы (ВКР) на тему: **«Разработка методики миграции и рефакторинга высоконагруженного веб-приложения на Vue.js с переходом к современной архитектуре на основе композиционных функций»**.

## 📁 Структура проекта

В рамках работы проведен сравнительный анализ двух подходов к разработке на примере реального компонента системы (анонимизированный код).

- 📂 `src/SingleComponent.vue` — Реализация **"ДО"** (Vue 2). Монолитный компонент на Options API. Логика, состояние и представление тесно переплетены.
- 📂 `src/MigratedSingleComponent/` — Реализация **"ПОСЛЕ"** (Vue 3). Модульная архитектура на Composition API.
  - `MigratedSingleComponent.vue` — "Тонкий" компонент представления (View).
  - `composables/MigratedSingleComponent.js` — Инкапсулированная бизнес-логика (Logic).
- 📂 `tests/` — Сравнительные модульные тесты.

## Запуск проекта и тестов

1. Установка зависимостей
Для работы проекта и тестов необходимо установить пакеты:

```bash
npm install
```

2. Запуск тестов
Для демонстрации преимуществ новой архитектуры (изолированное тестирование) выполните:

```bash
npm test
```

Ожидаемый результат: успешное прохождение тестов для MigratedSingleComponent без необходимости монтирования тяжелых компонентов.

📊 Результаты миграции и метрики
В ходе миграции компонента SingleComponent в архитектуру Vue 3 были получены следующие улучшения:

## Архитектурные изменения

| Характеристика | Vue 2 (Options API) | Vue 3 (Composition API) |
|----------------|---------------------|-------------------------|
| Разделение ответственности | Смешение UI и логики в одном файле | Полное разделение (View + Composable) |
| Переиспользование логики | Сложно (требует Mixins или дублирования) | Легко (импорт функции `use...`) |
| Типизация | Сложно (проблемы с типизацией `this`) | Нативная поддержка TypeScript |
| Реактивность | Глубокая (Observer) — избыточна для списков | Оптимизированная (`shallowRef`) |

## Количественные метрики
Сравнение проводилось на анонимизированном коде модуля работы со списками.

| Метрика | Vue 2 (SingleComponent) | Vue 3 (Migrated) | Динамика |
|---------|------------------------|------------------|----------|
| Объем компонента (.vue) | 241 строка | 160 строк | Сокращение на 34% |
| Объем логики (JS) | 123 строки (внутри .vue) | ~85 строк (отдельный .js) | Вынос в изолированный модуль |
| Количество файлов | 1 (монолит) | 2 (view + logic) | Разделение ответственности и упрощение поддержки |
| Объем кода тестов | 45 строки | 27 строк | Сокращение более чем в 1.5 раза |
| Зависимости теста | Требует mount, 12 заглушек (stubs), моков (mocks) | Требует только мок API | Упрощение изоляции |
| Тип тестирования | Интеграционный (медленный) | Модульный (быстрый) | Ускорение прогона |
| Время выполнения теста | ~100мс | ~2мс | Разница более чем в 50 раз |

В ходе миграции компонента SingleComponent была применена стратегия извлечения бизнес-логики (Logic Extraction). Общий объем кода остался сопоставимым (243 строки против 245 строк суммарно), однако структура фундаментально изменилась:

Снижение когнитивной нагрузки: Файл представления (.vue) сократился со 243 до 160 строк (на 34%). Разработчику больше не нужно прокручивать длинный блок методов, чтобы добраться до верстки.
Разделение ответственности: Логика загрузки данных, пагинации и обработки ошибок была инкапсулирована в изолированную функцию useMigratedSingleComponentLogic. Это позволило использовать shallowRef для хранения массива записей, что критически важно для высоконагруженных компонентов, так как снижает накладные расходы на инициализацию глубокой реактивности Vue 2.

Наиболее значимый результат был получен в области тестирования.

Для Vue 2 (Options API): Написание модульного теста потребовало использования метода mount из библиотеки @vue/test-utils. Это привело к необходимости создавать 12 заглушек (stubs) для дочерних UI-компонентов и мокать глобальные объекты ($t, $style), не относящиеся к логике. Тест получился громоздким (~45 строк), хрупким (зависит от структуры шаблона) и медленным (требует рендеринга виртуального DOM).

Для Vue 3 (Composition API): Благодаря выносу логики в composable-функцию, удалось провести чистое модульное тестирование (Unit Testing). Тест вызывает функцию напрямую как обычную JavaScript-процедуру.

Объем кода теста сократился более чем в 2 раза.
Исчезла зависимость от UI-компонентов и верстки.
Время выполнения теста сократилось с ~100мс (рендеринг) до ~2мс (чистый JS).
Это доказывает, что миграция на Composition API повышает тестируемость кода (Testability) и снижает трудозатраты на написание тестов.

## Качество тестирования
Vue 2 (Integration Testing):
Тестирование требовало создания полноценного экземпляра компонента (mount), мокания глобальных объектов (this.$t, this.$route) и работы с оберткой wrapper.vm.

❌ Медленно: Рендеринг виртуального DOM.
❌ Хрупко: Зависимость от верстки и дочерних компонентов.
Vue 3 (Unit Testing):
Благодаря Composition API мы тестируем чистую JavaScript-функцию MigratedSingleComponent изолированно.

✅ Скорость: Тесты выполняются за миллисекунды.
✅ Изоляция: Не нужно рендерить DOM для проверки бизнес-логики.
✅ Читаемость: Тест выглядит как проверка обычной функции.
